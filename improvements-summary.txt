HIREVERSE — ROBUSTNESS & ACCURACY IMPROVEMENTS
================================================
Date: 2026-06-11
Scope: Resume parsing, experience extraction, phone extraction, skill
matching/relations, interview-guide question bank, interview-feedback
analysis, and global search — all changes are additive (new patterns/
entries/branches) so existing behavior is preserved and nothing that
worked before regresses.

Verified: `npx astro check` produces the exact same 130 pre-existing
errors before and after these changes (all unrelated .astro DOM-typing
issues / a missing `word-extractor` type declaration) — zero new errors
introduced. Spot-tested against the three sample resumes in
src/fixtures/resumes/ and synthetic edge cases.


1. RESUME EXPERIENCE EXTRACTION — biggest fix (src/lib/resumeAnalysis.ts)
--------------------------------------------------------------------------
LOOPHOLE: extractExperience() only recognized date ranges written with a
month NAME ("Apr 2021 – Present", "Jun 2020 – Mar 2021"). Resumes that use
the extremely common numeric "MM/YYYY" or "MM-YYYY" format (e.g.
"03/2021 - 06/2023", common in LinkedIn-exported and templated resumes)
matched NOTHING — these candidates silently scored 0 years of experience,
which then tanks experienceMatch and the overall score even when the
resume clearly shows years of real work.

FIX:
 - Added a NUM_MONTH pattern and rebuilt the date-range regex (DATE_TOKEN)
   so each end of a range can be:
     • "15 Mar 2020" / "Mar 2020"  (month name, as before)
     • "03/2020" / "03-2020" / "03.2020"  (numeric month — NEW)
     • "2020"  (bare year, treated as Jan–Dec as before)
 - Open-ended ranges now also recognize "ongoing", "till date", "till now",
   "to date" in addition to the existing "present"/"current"/"now".
 - Verified: "03/2021 - 06/2023" + "01/2019 - 02/2021" now correctly sums
   to ~4.33 years (previously: 0). Existing month-name resumes (Aswani,
   Basudha, Muskan fixtures) produce identical results to before.


2. PHONE NUMBER EXTRACTION (src/lib/resumeAnalysis.ts)
--------------------------------------------------------------------------
LOOPHOLE: extractPhone() scanned the whole resume for any 10-13 digit
number-shaped string. On resumes that also contain employee IDs, postal
codes, or other long digit runs, the wrong number could be picked up
before the real phone number.

FIX: Added a labeled-phone pass that runs first — looks for
"Phone:", "Mobile:", "Mobile No:", "Cell:", "Contact:", "Tel:",
"Telephone:", "WhatsApp:" followed by a number, and uses that if it's a
valid 10-13 digit number. Falls back to the existing whole-text scan
unchanged if no labeled number is found — so resumes without a label
behave exactly as before.

Example: a JD requiring "Kubernetes" with a resume that only mentions

5. INTERVIEW FEEDBACK ANALYSIS LEXICON (src/lib/interviewAnalysis.ts)
--------------------------------------------------------------------------
The scoring engine already explicitly flags when feedback is "thin" and
its pattern library doesn't recognize the phrasing used — these additions
close several real gaps in that library without changing any existing
pattern's weight or behavior:

 - VERDICT signals: recognize quick scorecard shorthand "no hire" /
   "thumbs down" / "hard pass" / "strong no" (negative, weight 4.5) and
   "strong yes/hire" / "thumbs up" / "definite hire" / "easy yes"
   (positive, weight 4) — these are extremely common in real interview
   scorecards but didn't match any existing phrase.
 - TECHNICAL signals: recognize the interviewer's own third-person framing
   of a gap — "candidate has limited exposure to X" / "isn't familiar
   with Y" — mirroring the first-person uncertainty admissions already
   detected in transcripts.
 - First-person uncertainty admissions (transcript scanning) extended to
   also catch:
     • "I'm not really familiar with..."
     • "I'd have to look that up" / "I'd need to google that"
     • "I never really got a chance to..." / "I haven't had the
       opportunity to..."
   (Still only weighed in once 3+ such admissions appear in one
   conversation — a single honest "I don't know" remains healthy
   self-awareness, not a red flag, per the existing design.)


6. GLOBAL SEARCH (src/pages/api/search.ts)
--------------------------------------------------------------------------
LOOPHOLE: The top-bar search only matched job title/department/location
and candidate name/email/location — searching for a skill (e.g. typing
"Kubernetes" or "React") returned nothing even though that's one of the
most useful things a recruiter would search for.

FIX: Extended both $or queries (additive fields only):
 - Jobs: now also matches against requiredSkills and niceToHaveSkills.
 - Candidates: now also matches against their parsed skills list and
   recommendation label (e.g. "Strongly Recommend").


WHAT WAS DELIBERATELY NOT CHANGED
--------------------------------------------------------------------------
 - Name/location extraction heuristics: already extremely well-tuned with
   many documented edge cases (consultancy templates, contact-block
   parsing, etc.) — further regex changes here carry high regression risk
   for low marginal benefit without a real test corpus.
 - Email extraction: left as-is; obfuscated formats ("name [at] domain
   [dot] com") are rare enough on professional resumes that loosening the
   regex risks false positives elsewhere.
 - Full DD/MM/YYYY (with day) date ranges: numeric MM/YYYY is now
   supported (the common case); adding day-level numeric dates would
   require reworking the DAY token in a way that risks colliding with
   phone-number-shaped text, for a format rarely used in "experience"
   date ranges.

NEXT STEPS / RECOMMENDATIONS
--------------------------------------------------------------------------
 - Consider adding a small, checked-in test suite (e.g. vitest) covering
   extractExperience, extractPhone, matchSkills, and analyzeInterview with
   a handful of real anonymized resumes/feedback snippets — this would let
   future regex tightening be verified automatically rather than by
   spot-checking, and is the single highest-leverage next investment for
   "without causing regression" going forward.
 - The skill-question bank and skill-relations map are designed to be
   extended incrementally — as new JD skills come in that fall back to
   generic questions or score a hard "missing" with no related-skill
   credit, add them following the same pattern used here.

PART 2 — STAGE-CHANGE FLOW: BUG FIXES & SAMPLE DATA REFRESH

Date: 2026-06-11
Scope: candidate stage-transition logic (src/lib/pipeline.ts,
src/pages/api/candidates/[id].ts), the two UI surfaces that move a
candidate's stage (src/pages/candidates/[id].astro and
.../[id]/journey.astro), plus a full data refresh with diverse sample

1. INTERVIEW-GATE DETECTION MISSED NON-LITERAL STAGE NAMES (src/lib/pipeline.ts)
--------------------------------------------------------------------------
LOOPHOLE: findInterviewGateStage() only matched stages whose key/label
literally contained the substring "interview" (/interview/i). But the
"Schedule Interview" button (hasUpcomingInterviewStage / isInterviewStage)
already recognizes a much broader set of human-meeting stage names —
"Tech Round 1", "HR Round", "Panel Discussion", "System Design", etc. For
any job whose pipeline uses these (extremely common) names instead of the
literal word "Interview", findInterviewGateStage() returned undefined, so
the "block moving past the interview stage with no analyzed interview on
file" check NEVER fired for that job — a silent gap that defeats the whole
safeguard for non-default pipelines.

FIX: findInterviewGateStage() now reuses isInterviewStage() — the exact
same classification already used to decide whether "Schedule Interview"
should be offered — so the two stay consistent. Default pipeline
("Applied/Screening/Shortlisted/Interview/Offer/Hired") is unaffected
(its "Interview" stage matches both old and new logic identically).


2. GATE CHECK COULD BE SATISFIED BY A STALE EARLIER ROUND'S ANALYSIS
--------------------------------------------------------------------------
LOOPHOLE: The "moving past the interview stage requires a completed,
analyzed interview" check (NO_ANALYZED_INTERVIEW) only verified that *some*
analyzed interview existed for the candidate, anywhere — `Interview.exists({
candidateId, status: 'completed', analysis: {$ne: null} })`. For a pipeline
with multiple interview-type rounds (e.g. "Technical Interview" -> "Final
Interview" -> "Offer"), a candidate could be moved from "Final Interview"
straight to "Offer" using only the *Technical* interview's old analysis —
the Final Interview itself could be totally unanalyzed (or not even
scheduled) and the move would sail through silently.

FIX: if the stage the candidate is LEAVING is itself an interview-type
stage (isInterviewStage(fromStage)), the analyzed-interview lookup is now
scoped to that specific stage (`pipelineStage: fromStage.key`) — so the
round being left behind must itself have a completed, analyzed record.
Moving across the gate for the first time (from a non-interview stage)
keeps the original "any analyzed interview exists" check, unchanged.


3. NO ANALOGOUS GATE FOR ASSESSMENT STAGES (coding tests / take-homes)
--------------------------------------------------------------------------
LOOPHOLE: isAssessmentStage()/hasUpcomingAssessmentStage() already exist
and drive the "Create Assessment" action, but there was no equivalent to
the interview gate for assessment stages — a candidate could be moved past
a "Coding Test" or "Take-Home Assignment" stage with the assessment still
`pending`/`submitted` and never `evaluated`, leaving their AI score and
recommendation silently out of step with a journey that says they cleared
a coding test they never actually completed.

FIX: Added findAssessmentGateStage() (mirrors findInterviewGateStage, using
isAssessmentStage) and a matching NO_ANALYZED_ASSESSMENT 409 check in the
PUT handler — same shape as the interview one (requiresConfirmation: true,
override via confirmSkipAssessment: true), scoped to the stage being left
when that stage is itself assessment-type, otherwise "any evaluated+
analyzed assessment exists".


4. "MOVE ANYWAY" OVERRIDE WAS UNREACHABLE FROM THE MAIN CANDIDATE PAGE
--------------------------------------------------------------------------
LOOPHOLE: The backend's NO_ANALYZED_INTERVIEW response was designed as a
"soft block" — `requiresConfirmation: true` plus a `confirmSkipInterview`
override flag. The candidate journey page (.../[id]/journey.astro) already
wired this up correctly with a confirm dialog. But the main candidate
detail page (candidates/[id].astro), which has its own stage <select>, did
NOT — it just toasted the raw error message and reverted the dropdown,
with no way to proceed. A recruiter hitting this on the main profile page
had to know to go to the Journey page instead, which isn't discoverable.

FIX: candidates/[id].astro's stage-change handler now mirrors the journey
page: on a 409 with requiresConfirmation, it shows a "Move anyway?" confirm
dialog with the backend's explanation and, if confirmed, retries the same
PUT with confirmSkipInterview/confirmSkipAssessment (chosen based on the
returned `code`). The journey page's handler was generalized the same way
to also handle the new NO_ANALYZED_ASSESSMENT code.


5. SCHEDULED ASSESSMENTS DIDN'T FOLLOW A CANDIDATE WHO MOVED ON
--------------------------------------------------------------------------
LOOPHOLE: When a candidate's stage changes, any still-`scheduled` Interview
pinned to the stage they're leaving is moved forward to their new stage
(so it doesn't get stranded under a stage they've left). Assessments had no
equivalent — a `pending`/`submitted` Assessment stayed pinned to the old
`pipelineStage` forever, becoming invisible/orphaned on the journey view
once the candidate moved past that stage.

FIX: Added the same forward-following update for Assessments
(`status !== 'evaluated'`) alongside the existing Interview one, in the
same `if (fromStage)` block.


6. NO WAY TO UNDO A MIS-CLICKED "REJECT CANDIDATE"
--------------------------------------------------------------------------
LOOPHOLE: PUT /api/candidates/[id] only accepted `rejected: true` (a
one-way terminal action); `rejected: false` was silently stripped by the
generic-update field whitelist. The UI's reject confirmation dialog said
"This is a terminal action and cannot be undone" — true, but a single
mis-click (or a candidate who reapplies/gets a second look) had no recovery
path short of editing the database directly.

FIX: Added an explicit "undo rejection" branch — `rejected: false` on a
currently-rejected candidate clears `rejected`/`rejectedAt`/`rejectedBy`,
logs a `candidate_unrejected` activity entry, and returns the updated
candidate (400 if the candidate isn't currently rejected). Both the main
candidate page and the journey page now show an "Undo Rejection" button
when `candidate.rejected` is true, and the reject dialog's copy was updated
to "This can be undone later from this page if needed."


6b. ORPHANED ASSESSMENTS ON CANDIDATE DELETE (src/pages/api/candidates/[id].ts)
--------------------------------------------------------------------------
LOOPHOLE: DELETE /api/candidates/[id] removed the candidate's Interviews but
not their Assessments — deleting a candidate left orphaned Assessment
documents pointing at a non-existent candidateId.

FIX: Added `Assessment.deleteMany({ candidateId: id })` alongside the
existing Interview cleanup.


7. "REMOTE (CITY, COUNTRY)" RESUME HEADERS WERE INVISIBLE TO LOCATION EXTRACTION
   (src/lib/resumeAnalysis.ts)
--------------------------------------------------------------------------
LOOPHOLE: Found while building the new sample DevOps Engineer resume, which
(realistically, for a remote role) heads with "Remote (Gurgaon, India)".
The location regexes (LOCATION_STRICT/LOOSE, both anchored "City, Region"
patterns) don't allow the leading word "Remote" or the wrapping
parentheses, so this extremely common modern remote-work resume header
matched nothing — the candidate came back as "Location Not Found" / "none"
confidence even though their city is right there on the page.

FIX: Added stripRemotePrefix(), applied wherever a location-shaped line is
tested (the contact-block "|"-segment scan, the near-contact-anchor scan,
and the main per-line scan). It strips a leading "Remote" qualifier in any
of its common forms — "Remote (Gurgaon, India)", "Remote - Gurgaon, India",
"Remote: Gurgaon, India", "Remote, Gurgaon, India" — leaving "Gurgaon,
India" to match LOCATION_LOOSE/STRICT as before. Lines that don't start
with "Remote" are returned unchanged, so existing extraction is untouched
(spot-checked against the three original fixtures — identical results).

3. "SCHEDULE INTERVIEW" / "CREATE ASSESSMENT": BOTH SHOWING AT ONCE
   (src/lib/pipeline.ts, src/pages/candidates/[id].astro,
   src/pages/candidates/[id]/journey.astro)
--------------------------------------------------------------------------
BUG: A candidate's profile could show both "Schedule Interview" and
"Create Assessment" buttons simultaneously, even though only one stage is
actually "next" for them. This happened because
hasUpcomingInterviewStage()/hasUpcomingAssessmentStage() (used to compute
canScheduleInterview/canScheduleAssessment) check whether ANY stage from
the candidate's current position to the END of the pipeline matches that
type — so a pipeline like "Resume Screen -> Coding Test -> Technical
Interview -> Offer -> Hired" would show BOTH buttons for a candidate who is
still at "Resume Screen", because both an assessment stage and an interview
stage exist somewhere ahead.

WHAT THE USER WANTS: only the action relevant to the candidate's actual
current stage should show. If they're at an assessment-type stage and
haven't completed it yet, only "Create Assessment" should show. Once that
assessment is evaluated, "Create Assessment" should disappear and, if the
next stage is interview-type, "Schedule Interview" should appear instead —
and vice versa.

FIX: New getCandidateStageActions(pipeline, currentStageKey,
interviewStatusForStage, assessmentStatusForStage) in src/lib/pipeline.ts —
the single source of truth for which one action (if any) to show:
 - Looks at the candidate's CURRENT stage first. If it's an interview-type
   stage:
     - no interview record for that stage yet -> show "Schedule Interview"
       only
     - an interview record exists but isn't 'completed' yet (scheduled/
       rescheduled) -> show NEITHER button (nothing to do until that
       session happens or is cancelled)
     - a 'completed' interview exists for that stage -> the stage is
       "done"; fall through to check the NEXT stage
   Same logic for an assessment-type current stage, using 'evaluated' as
   the "done" status and "Create Assessment" as the action.
 - If the current stage isn't an interview/assessment stage (e.g. "Resume
   Screen", "Portfolio Review", "Offer"), or its gate is already satisfied,
   the function looks at the NEXT stage in the pipeline the same way — so
   "Resume Screen -> Coding Test -> Technical Interview" shows "Create
   Assessment" while at "Resume Screen", then (once the Coding Test is
   evaluated) "Schedule Interview" becomes the one shown, even before the
   candidate's currentStage has actually moved to "Technical Interview".
 - A 'cancelled' interview is treated as if it never happened (status
   'none'), so "Schedule Interview" reappears rather than getting stuck
   behind a dead record.
 - Returns { canScheduleInterview, canScheduleAssessment } — both false
   once the pipeline has no more actionable interview/assessment stages
   ahead (e.g. candidate is between "Technical Interview" (done) and
   "Offer", with nothing left to schedule/create).

Applied in both src/pages/candidates/[id].astro and
src/pages/candidates/[id]/journey.astro, replacing the old
hasUpcomingInterviewStage/hasUpcomingAssessmentStage-based checks. Each page
builds interviewStatusForStage/assessmentStatusForStage from the
Interview/Assessment documents it already loads (interviewDocs/
assessmentDocs in [id].astro, the existing interviewsByStage/
assessmentsByStage maps in journey.astro) — no new queries added.

NOT CHANGED: hasUpcomingInterviewStage/hasUpcomingAssessmentStage themselves
are left as-is and still used by src/pages/interviews/schedule.astro,
src/pages/assessments/create.astro, and src/pages/interviews/[id].astro,
which answer a different question ("should this candidate appear at all in
the pick-a-candidate list for scheduling/creating") — changing those was out
of scope and risked regressions on pages not mentioned in this request.

BUG FOUND AND FIXED (exactly the kind of robustness issue this seed was
meant to surface):
 - Arjun's Markdown resume starts with `# Arjun Mehta` and a
   `**Location:** Hyderabad, India` line. Before the fix, this came out as
   candidate name "Arjun Mehta.md" (the .md extension leaked from the
   filename-fallback path) with location "Location Not Found".
 - Root cause #1 (src/lib/resumeAnalysis.ts, extractName): the line-by-line
   name heuristic saw "# Arjun Mehta" as a 3-word line ("#", "Arjun",
   "Mehta") that didn't "look like a name" because of the leading "#", so
   it fell through every heuristic to extractNameFromFilename — whose
   extension stripper only recognized .pdf/.docx/.doc/.txt, not .md,
   leaving ".md" stuck onto "Mehta".

     a) analyzeResume() now strips common Markdown markup
        (`**bold**`, `__bold__`, leading `#`/`##`/... headers, leading
        `-`/`*`/`+` bullet markers) from the text up front, before any
        field extraction runs — so "# Arjun Mehta" reads as "Arjun Mehta"
        and "**Location:** Hyderabad, India" reads as "Location: Hyderabad,
        India" (which the existing LOCATION_LABELED regex already handles).
     b) extractNameFromFilename()'s extension stripper was generalized from
        `/\.(pdf|docx|doc|txt)$/i` to `/\.[a-zA-Z0-9]{1,5}$/` so ANY short
        file extension (.md, .rtf, .odt, etc.) is stripped, not just the
        four originally hard-coded ones.
 - Verified after the fix: Arjun's resume now extracts name "Arjun Mehta",
   location "Hyderabad, India" (medium confidence), and the existing
   "Arjun Mehta.md" candidate record in the database was corrected to
   "Arjun Mehta".
 - Also fixed two pre-existing relative-import bugs that blocked running
   any seed script with `node --experimental-strip-types` (missing `.ts`
   extensions in src/lib/codingChallenges.ts's import of skillQuestions and
   src/lib/activity.ts's import of ActivityLog) — harmless for
   Astro/Vite, required for direct Node execution.

PART 7 — ADAPTIVE INTELLIGENCE LAYER (PLATFORM-WIDE LEARNING, NO REGRESSION)
================================================================================

GOAL: implement the "every piece of data feeds back in and makes the platform
smarter, forever" capability discussed with senior management — without any
black-box ML, without ever letting a score get WORSE, and without changing a
single existing code path's output when no learning signal exists yet (fresh
install = byte-identical behavior to before).

Two independent, additive, fully-auditable mechanisms, both designed around
ai-architecture-recommendation.txt section 10 (options 1 and 2):

1. SKILL ALIAS LEARNING (src/lib/learningEngine.ts, recordSkillCorrection /
   getLearnedAliases)
 - New models: src/db/models/LearningSignal.ts (append-only event log) and
   src/db/models/SkillIntelligence.ts (per-skill alias tally).
 - On a candidate's page, any required/nice-to-have skill that scored
   "missing" (0%) now shows a "Found evidence on resume?" link in the Skills
   Analysis card. A recruiter quotes the exact word/phrase from the resume
   that demonstrates real knowledge of that skill.
 - New API route: src/pages/api/candidates/[id]/skill-correction.ts
   - Verifies the phrase actually appears in the candidate's resume text
     (rejects fabricated corrections).
   - Only allowed from "missing" status — "related" (35%) is strictly higher
     than "missing" (0%), so this can never make a score worse. "listed"
     (65%) and "practical" (100%) are already above "related" and aren't
     eligible for correction.
   - Re-scores the candidate via the new shared
     `recomputeFromSkillGaps()` (src/lib/resumeAnalysis.ts) — the exact same
     formula `analyzeResume()` uses, so there's zero drift between the two
     code paths.
   - Records a LearningSignal and bumps the (skill, phrase) tally in
     SkillIntelligence.
 - Once the SAME (skill, phrase) pair has been confirmed by
   LEARNED_ALIAS_THRESHOLD (=2) independent corrections, it's "promoted":
   src/lib/skillRelations.ts's new `findLearnedEvidence()` then recognizes
   that phrase as evidence of that skill on EVERY future resume, for ANY job
   that requires it — exactly like the existing static SKILL_RELATIONS
   lexicon, but grown from real recruiter judgment instead of being
   hand-curated.
 - Wired into both src/pages/api/candidates/upload.ts and
   src/pages/api/candidates/[id]/reanalyze.ts via
   `getLearnedAliases()` -> `analyzeResume(..., learnedAliases)`. On a fresh
   install `getLearnedAliases()` returns `{}`, which makes
   `findLearnedEvidence` a no-op — identical output to before this layer
   existed.
 - Candidate.ts: skillGaps entries gained an optional `learnedOccurrences`
   field so the UI can label a learned-alias match as "(learned from N
   recruiter corrections)".

2. OUTCOME CALIBRATION (src/lib/learningEngine.ts, recordOutcomeSignal /
   summarizeOutcomeSignals)
 - Every time a candidate is rejected or moves to a new pipeline stage
   (src/pages/api/candidates/[id].ts PUT), a LearningSignal is recorded with
   their resume score at that moment and the outcome ('advanced', 'hired', or
   'rejected'). `isHiredStage()` (existing pipeline helper) decides hired vs
   advanced.
 - This is purely descriptive and append-only — it is NEVER read back into
   any candidate's own score. It only feeds the aggregate insight below.
 - `summarizeOutcomeSignals()` buckets signals by job and by the same score
   bands as getRecommendation (0-64 / 65-74 / 75-87 / 88-100), showing the
   real advance/hire/reject rate for each band — i.e. whether the scoring
   rubric for that specific role is actually predictive.

3. NEW PAGE: /intelligence (src/pages/intelligence/index.astro), added to the
   sidebar nav (src/layouts/AppLayout.astro) as "Intelligence".
 - Top stat cards: total skill corrections recorded, aliases now Active vs
   still Learning (out of LEARNED_ALIAS_THRESHOLD), total outcome signals.
 - "Learned Skill Aliases" list — every (skill, phrase) pair with its
   confirmation count and Active/Learning badge.
 - "Score Calibration by Job" — per-job score-band vs outcome breakdown,

PART 10 — STAGE-SKIP GUARDRAIL, THEME-FLICKER FIX, SCROLLBAR/STICKY-HEADER
PASS, AND ANALYTICS FILTERS + CSV EXPORT

This pass addressed four reports from a single review session: (1) the
candidate-profile stage dropdown let anyone jump a candidate straight to any
later stage by adding one comment, with no warning that intermediate stages
(and their interviews/assessments) would be skipped; (2) toggling theme or
navigating between tabs produced a visible flash of the wrong theme's colors;
(3) several growing lists/tables had no scroll containment, so they grew the
page indefinitely; (4) the Analytics page had no way to filter by job or time
range, and no way to export the underlying data.

1. STAGE-SKIP CONFIRMATION (src/pages/candidates/[id].astro)
 - Root cause: `getValidNextStages()` (src/lib/pipeline.ts) returns the
   candidate's current stage plus EVERY later stage in the pipeline — by
   design, so recruiters can fast-track a strong candidate. The API
   (`isValidStageTransition`) only rejects BACKWARD moves, and per-stage
   gate checks (`confirmSkipInterview`/`confirmSkipAssessment`) only fire for
   stages explicitly flagged as interview/assessment gates. Combined, the UI
   let a single stage-change + note silently skip any number of stages with
   no indication anything unusual had happened.
 - Fix (additive, server-side gates untouched):
   - The stage `<select>` now carries `data-current-order` and a per-option
     `data-order`, and the option immediately after the current stage is
     labeled "(Next)" so the natural, single-step move is visually obvious.
   - On change, the page now computes which stages (if any) sit strictly
     between the current stage and the selected stage. If any are skipped,
     `window.hvConfirm()` shows a named, destructive-styled confirmation:
     "This skips N stage(s) the candidate hasn't been through yet: <names>.
     Any interview or assessment for those stages won't be scheduled.
     Continue?" Declining reverts the dropdown to the current stage and the
     existing note-entry flow (`hvPrompt`) never opens.
   - Accepting proceeds into the existing note-entry and PATCH flow exactly
     as before — `confirmSkipInterview`/`confirmSkipAssessment` gate logic in
     `src/pages/api/candidates/[id].ts` is unchanged.
 - Net effect: single-step and "Next"-stage moves are unaffected; any
   multi-stage jump now requires an explicit, informed, named confirmation.

2. THEME-TOGGLE / PAGE-NAVIGATION FLICKER FIX (src/layouts/AppLayout.astro)
 - Root cause: HireVerse is a multi-page app — every sidebar click is a full
   navigation, and `<html>` hardcodes `class="dark"`. The previous
   light/dark detection lived in a regular `<script>` tag, which Astro
   bundles as a deferred ES module that runs AFTER first paint. Light-theme
   users therefore saw every page flash dark, then snap to light.
 - Fix: added a `<script is:inline>` block at the end of `<head>` (runs
   synchronously, before first paint, on every navigation) that removes the
   `dark` class immediately if `localStorage.getItem("hv-theme") ===
   "light"`. The old detection IIFE was removed from the deferred script,
   which now only wires up the toggle button/icon — its logic is unchanged.
 - In-page color transitions when toggling the theme (without navigating)
   were already smooth via the existing global `transition-property:
   background-color, border-color, color, fill, stroke` rule in
   src/styles/global.css — that rule was left as-is.

3. SCROLLBAR / STICKY-HEADER PASS (jobs, team, interviews, assessments,
   candidates, job-detail pages)
