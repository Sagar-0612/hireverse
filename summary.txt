HireVerse — ATS Reliability/UX Fixes (Rounds 1, 2 & 3)
========================================================
All changes are deterministic / heuristic (no LLM, no external AI API),
consistent with the existing resumeAnalysis.ts / interviewAnalysis.ts /
assessmentAnalysis.ts style.


1. JOB TITLE DUPLICATE DETECTION
---------------------------------
Problem: Two jobs could be created with the same (or whitespace/case-different)
title with no warning.

Changes:
- New src/lib/text.ts: normalizeTitle(s) — trims, collapses internal
  whitespace, and lowercases a title for comparison.
- src/pages/api/jobs/index.ts (POST): before creating a job, all existing job
  titles are normalized and compared against the incoming title (case- and
  whitespace-insensitive, across ALL jobs regardless of status). A match
  returns 409 with: 'A job titled "<title>" already exists. Choose a
  different title.'
- src/pages/api/jobs/[id].ts (PUT): same check when a title is edited,
  excluding the job's own _id.
- src/pages/jobs/new.astro: on failure, the error is now also shown as a
  toast (window.hvToast) in addition to the existing error banner.


2. GAP-FOCUSED, ACCUMULATING INTERVIEW GUIDE — WITH REAL QUESTIONS
-------------------------------------------------------------------
Problem: The "Interview Guide" was a generic 6-step checklist, identical for
every candidate and every round, and (round 1) only described gaps without
giving interviewers anything concrete to ask.

Changes:
- src/lib/interviewGuide.ts:
  - GuideStep now has an optional `questions: string[]` field — concrete,
    ready-to-ask interview questions, not just a description of the gap.
  - generateFocusAreas(input) builds focus areas from:
      * the candidate's skillGaps (missing / related / listed skills, ordered
        by severity, capped at 5). Each gets 2 tailored questions:
          - missing:  "Have you worked with <skill> in any capacity — even a
                       small project, course, or personal experiment?" /
                       "If you had to start using <skill> on the job next
                       week, how would you ramp up?"
          - related:  "Your resume highlights <relatedSkill> — can you walk
                       me through how that work involved (or could extend
                       to) <skill>?" / "What's the most complex thing you've
                       done with <skill> specifically, separate from
                       <relatedSkill>?"
          - listed:   "You've listed <skill> on your resume — talk me
                       through a specific project where you used it and what
                       your role was." / "What's the trickiest problem you've
                       solved using <skill>?"
      * an education-gap item (with 2 questions) if educationMatch < 65
      * an experience-gap item (with 2 questions) if years of experience fall
        below the job level's expected range (parseLevelRange /
        inferLevelYears, exported from resumeAnalysis.ts)
      * "carry-forward" items — concerns from any prior completed
        interview/assessment's analysis.concerns, each with a direct
        follow-up question quoting the original concern, so round 2's guide
        includes round 1's unresolved concerns, etc.
  - generateInterviewGuide(input) wraps generateFocusAreas with a standard
    intro/behavioral/Q&A/wrap-up structure (behavioral round also now ships
    with 2 real questions), used for the numbered "Interview Guide" steps.
- src/pages/interviews/[id].astro:
  - jobId populate includes requiredSkills, niceToHaveSkills, education,
    level.
  - candidate fetch includes skillGaps, score, skillsMatch, educationMatch,
    experience, recommendation.
  - Fetches all other completed interviews (with analysis) and evaluated
    assessments (with analysis) for the candidate.
  - The hardcoded 6-step guide is replaced with generateInterviewGuide()
    output: each numbered step shows a bold title, a detail line, and (when
    present) a bulleted list of the actual questions to ask.
- src/pages/assessments/[id].astro: for pending/submitted assessments, the
  "Suggested Focus Areas" card (using generateFocusAreas, same data sources)
  now also renders each focus area's question list.


3. SCHEDULING NOTIFICATIONS (EMAIL + WHATSAPP)
-------------------------------------------------
Problem: Scheduling an interview/assessment sent no notifications to anyone.

Changes:
- src/lib/notifications.ts:
  - sendEmail({to, subject, text}) — uses nodemailer with SMTP_HOST/PORT/
    USER/PASS/FROM env vars. If SMTP_HOST is unset, logs
    "[email:not-configured] ..." to the console and returns {sent:false} —
    never throws.
  - sendWhatsApp({to, message}) — uses Twilio's REST API directly via fetch
    (TWILIO_ACCOUNT_SID/AUTH_TOKEN/WHATSAPP_FROM env vars, no SDK needed). If
    unset, logs "[whatsapp:not-configured] ..." and returns {sent:false}.
- src/pages/api/interviews/index.ts (POST): after the interview is created
  and the scheduling activity is logged (non-blocking try/catch):
  - Candidate gets an email AND a WhatsApp message with round, job, date,
    time, duration, format, and interviewer.
  - Interviewer (looked up via Team.findOne({name: interviewer})) gets an
    email with the candidate's name and the same details.
  - A new ActivityLog entry ("interview_notifications_sent") records what was
    actually sent vs. skipped (missing contact info / unconfigured).
- src/pages/api/assessments/index.ts (POST): same pattern — candidate gets an
  email + WhatsApp message announcing the new assessment (with instructions
  if provided), logged the same way.
- package.json: added "nodemailer" (dependency) and "@types/nodemailer"
  (devDependency); also added @astrojs/check + typescript (required by
  `astro check`, were missing from devDependencies).
- .env.example: documented optional SMTP_* and TWILIO_* env vars, with notes
  that the system works (log-only) without them.

STATUS — VERIFIED WORKING AS DESIGNED:
This codebase's .env currently has only MONGODB_URI set — no SMTP_HOST or
TWILIO_* credentials. We confirmed (via direct DB inspection) that scheduling
"Technical Round 1" for Sagar Pawar correctly ran the notification code path
and logged an ActivityLog entry: "No notifications could be sent for
'Technical Round 1' (Sagar Pawar) — missing contact info or notification
config". The candidate's email/phone WERE present, so the only thing
preventing real delivery is the absence of SMTP/Twilio configuration. To
actually send emails and WhatsApp messages, add the following to .env:
  SMTP_HOST, SMTP_PORT, SMTP_USER, SMTP_PASS, SMTP_FROM
  TWILIO_ACCOUNT_SID, TWILIO_AUTH_TOKEN, TWILIO_WHATSAPP_FROM
Once set, sendEmail/sendWhatsApp automatically switch from logging to actually
sending — no further code changes needed.

5. RELATED-SKILL PARTIAL CREDIT (RESUME SCORING) — INCLUDING PLURAL FIX
--------------------------------------------------------------------------
Problem: A required skill not literally found in the resume scored 0, even
when a clearly related skill was present (e.g. resume shows "Node.js" but the
job requires "REST API" — no credit given). Worked the same for non-IT roles.

Round-1 implementation:
- New src/lib/skillRelations.ts:
  - SKILL_RELATIONS: a curated lowercase skill -> related-skills lexicon
    covering common IT domains (web/JS, databases, devops/cloud, python/data,
    Java, .NET/PHP/Ruby, mobile, QA, project management) and non-IT domains
    (sales/marketing, accounting, HR, customer support, design).
  - findRelatedEvidence(text, skill) — returns the first related skill found
    in the resume text (word-boundary matching), or null.
- src/lib/resumeAnalysis.ts:
  - New SkillGap type: { skill, status: 'practical'|'listed'|'related'|
    'missing', relatedSkill?, score }.
  - New scoring tiers: practical=100, listed=65, related=35 (NEW —
    previously 0), missing=0.
  - buildSkillGaps() classifies every required + nice-to-have skill into one
    of these 4 tiers (related uses findRelatedEvidence).
  - skillsMatch is now the weighted average of these per-skill scores
    (required skills weighted 70%, nice-to-have 30%).
  - analyzeResume() now returns skillGaps: SkillGap[].
  - parseLevelRange and inferLevelYears are now exported (reused by
    interviewGuide.ts for the experience-gap check).
- src/db/models/Candidate.ts: new ISkillGap subdocument schema and
  skillGaps: ISkillGap[] field on the Candidate model.
- src/pages/api/candidates/upload.ts, src/scripts/seed-sample-data.ts:
  persist skillGaps from analyzeResume() onto the candidate document (new
  candidates and re-applications).
- src/pages/candidates/[id].astro: the per-skill match display reads from
  candidate.skillGaps, showing 4 distinct levels:
    - "Used in a project or role — not just listed" (practical, ~100%)
    - "Listed, but no project/work evidence found" (listed, ~65%)
    - 'Not directly listed, but "<relatedSkill>" suggests working knowledge'
      (related, ~35%, amber bar)
    - "Not found on resume" (missing, 0%, red bar)

Round-2 bug fix — plural/singular skill-name mismatch:
- Found via testing: a job requiring "REST APIs" (plural) scored the related
  Node.js evidence as 0, because SKILL_RELATIONS' lexicon key is the singular
  "rest api" and normSkill() only lowercases + strips dots (no plural
  handling), so SKILL_RELATIONS['rest apis'] was undefined.
- Fixed in src/lib/skillRelations.ts:
  - findRelatedEvidence() now falls back to a singularized lexicon key:
    SKILL_RELATIONS[key] || SKILL_RELATIONS[key.replace(/s$/, '')].
  - appearsInText() now tolerates a trailing-"s" mismatch in either direction
    (only for terms >= 5 chars, to avoid false positives like "css" -> "cs").
- Fixed in src/lib/resumeAnalysis.ts:
  - New exported buildSkillRegex(skill) — shared word-boundary regex builder
    with the same trailing-"s" tolerance, used by matchSkills() and
    hasPracticalEvidence() so "REST API" <-> "REST APIs" match consistently
    everywhere skill text is matched.
- Verified: candidate Sagar Pawar (job "Full Stack Developer", which requires
  "REST APIs") now shows skillGaps: { skill: "REST APIs", status: "related",
  relatedSkill: "nodejs", score: 35 } — partial credit is correctly applied.

7. NO DUPLICATE INTERVIEW/ASSESSMENT FOR THE SAME PIPELINE STAGE

10. "EVALUATE ASSESSMENT" POPUP — FIXED PERMANENTLY-VISIBLE OVERLAY
-------------------------------------------------------------------------
Problem: On the assessment detail page, the "Evaluate Assessment" modal
("coding feedback" popup) was not "clearly visible" — in fact the underlying
page was obscured because the modal was rendering on top of it at all times.

Root cause: src/pages/assessments/[id].astro line 303 had a duplicated
`display` property in one inline style attribute:
  style="display:none; ... ; display:flex; ..."
In CSS, the later declaration wins for duplicate properties within the same
style attribute, so the modal's actual initial display was `flex` — fixed,
full-viewport, semi-transparent overlay — covering the page on every load,
not just when "Evaluate" was clicked.

Fix: removed the duplicate `display:flex`, leaving `display:none` as the
modal's initial state (JS openModal()/closeModal() toggle it as intended).
Checked the rest of src/pages/**/*.astro for the same duplicate-display
pattern — this was the only occurrence.

  getSkillQuestions() (was 2), giving 4 total questions per gap (1 bridging +
  3 topic, level-calibrated as before) — enough to properly probe a gap
  without being generic.
- `npx astro check` — no new errors/warnings introduced by any Round-3 change
